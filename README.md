# Backend Portfolio Pages

GitHub Pages에 바로 올릴 수 있도록 만든 정적 백엔드 포트폴리오 초안입니다.  
핵심 목표는 "신입 지원자"라는 라벨보다, `실무 감각이 있는 백엔드 주니어`로 읽히게 만드는 것입니다.

## 설계 방향

이 포트폴리오는 단순 소개 페이지가 아니라 아래 질문에 빠르게 답하도록 설계했습니다.

1. 어떤 백엔드 문제를 다뤄봤는가
2. 구조 변경과 수치로 설명 가능한가
3. 스타트업 환경에서 바로 써먹을 수 있는 감각이 있는가

그래서 페이지 흐름을 다음처럼 잡았습니다.

- `Hero`: 한 줄 포지셔닝과 핵심 증빙 수치
- `Career`: 실무 경험 요약
- `Projects`: 대표 프로젝트 2개
- `Engineering Cases`: 면접에서 꺼내기 좋은 기술 사례 카드
- `Workflow`: 이 포트폴리오를 어떤 기준으로 구성했는지 설명
- `Deploy`: 배포 전 교체할 값과 GitHub Pages 안내

## 포트폴리오 작성 워크플로우

### 1. 포지셔닝 고정

한 줄 소개는 아래 둘 중 하나로 고정하는 것이 좋습니다.

- `문제를 설계 경계로 다시 정의하는 백엔드 개발자`
- `정합성과 성능을 수치와 문서로 설명하는 백엔드 개발자`

이 문장은 이력서, GitHub 프로필, 포트폴리오 첫 화면에서 통일하는 편이 좋습니다.

### 2. 대표 사례 5~7개만 고정

모든 기술 문서를 다 메인에 올리지 말고, 아래처럼 면접에서 이야기하기 좋은 사례만 남기는 게 좋습니다.

- Hipster
  - 유저 신뢰도 계층 분리
  - 평점 집계 계층과 결과적 일관성
  - 차트 읽기 모델과 캐시 계층 분리
  - 차트 공개 파이프라인 분리
  - 정산 상태 모델링
- SNS
  - 피드 읽기 경로 재설계
  - 커넥션 풀보다 수평 확장

### 3. 각 사례는 같은 포맷으로 정리

각 카드나 문서는 아래 포맷을 유지하면 읽는 사람이 빠르게 이해합니다.

- 문제
- 판단 기준
- 구조 변경
- 수치 또는 검증 결과
- 남은 한계

### 4. 프로젝트 소개는 기능 목록이 아니라 문제 묶음으로 정리

좋은 예:

- 차트 시스템의 읽기 성능과 공개 정합성을 함께 다룬 프로젝트
- 피드 조회 비용을 어디에 둘지 설계한 프로젝트

피해야 할 예:

- 회원가입, 로그인, CRUD, 댓글, 좋아요 구현

### 5. 수치는 조건과 함께 적기

포트폴리오에 성능 수치를 적을 때는 아래를 같이 적는 편이 좋습니다.

- 로컬 기준인지, 테스트 환경 기준인지
- 데이터 규모가 어느 정도인지
- 운영 트래픽 대표값이 아니라는 점

이렇게 해야 오히려 신뢰도가 올라갑니다.

### 6. 스타트업 지원용 문장으로 정리

자기소개나 프로젝트 소개 문장은 아래 축으로 정리하면 좋습니다.

- 빠르게 구조를 세우고 검증할 수 있다
- 운영 중 설명 가능한 상태 모델을 선호한다
- 단순 기술 도입보다 병목의 원인을 분해해서 접근한다
- 문서화와 측정을 통해 팀이 이해할 수 있게 만든다

## 배포 워크플로우

### 추천 방식

가장 깔끔한 무료 배포는 `username.github.io` 사용자 페이지입니다.

현재 구조는 아래 가정을 두고 만들었습니다.

- GitHub username: `kangjoo2002`
- 배포 레포 이름: `kangjoo2002.github.io`

이 이름이 맞으면 사이트 주소는 아래처럼 됩니다.

- `https://kangjoo2002.github.io`

### 배포 순서

1. GitHub에서 `kangjoo2002.github.io` 레포를 새로 만듭니다.
2. 이 폴더 내용을 그 레포 루트에 올립니다.
3. 저장소 `Settings > Pages`에서 `GitHub Actions`를 선택합니다.
4. `main` 브랜치에 push 하면 `.github/workflows/deploy.yml`이 자동 배포합니다.

## 배포 전 반드시 교체할 값

현재는 빠르게 공개 가능한 초안을 만드는 데 집중했기 때문에 아래는 placeholder 성격이 남아 있습니다.

- 실명 또는 표시 이름
- 이메일
- 블로그 URL
- SNS 프로젝트 레포 URL
- Hipster 기간
- SNS 기간

## 파일 구성

- `index.html`: 메인 페이지
- `styles.css`: 디자인 시스템과 반응형 스타일
- `script.js`: 사례 필터, 섹션 하이라이트, reveal 애니메이션
- `.github/workflows/deploy.yml`: GitHub Pages 배포 workflow
- `.nojekyll`: 정적 파일 배포 안정화

## 다음으로 추천하는 수정 순서

1. 이름, 이메일, 블로그 링크를 채웁니다.
2. SNS 레포 URL이 있으면 연결합니다.
3. 각 프로젝트 기간을 입력합니다.
4. Hipster 대표 문서 1~2개에 영어 요약을 추가할지 판단합니다.
5. README와 이력서의 한 줄 포지셔닝 문구를 같은 문장으로 통일합니다.
